# Feature Requirements: QA Redesign — Executable Oracle + Adversarial Persona + Independent Planning

## Overview

Redesign the `documenting-qa` and `executing-qa` skills so that QA produces a real engineering signal instead of a structurally-forced PASS. The redesign has three layers: an **executable oracle** (the QA agent writes and runs real tests in the consumer repo's existing test framework and is graded on results, not on emitted phrases), **no-spec-during-planning** (the planning agent builds its plan from the user-facing summary and running code, not from the requirements doc that drove implementation), and an **adversarial tester persona** (the agent probes inputs, state transitions, environment failures, dependency failures, and cross-cutting concerns like a11y / i18n / concurrency rather than checking FR rows off a grid). Ships one generic "QA" persona; the skill composition is structured so additional personas (a11y, security, performance) can be added as follow-up features without restructuring the skill.

## Feature ID

`FEAT-018`

## GitHub Issue

[#170](https://github.com/lwndev/lwndev-marketplace/issues/170)

## Priority

High — Across 34 recorded QA runs in `qa/test-results/`, every single run returned `Verdict: PASS`. The QA skills run on every orchestrated workflow (each skill at least once) but produce no actionable output. The closed-loop planning + PASS-phrase stop hook + post-hoc execution position structurally forces that outcome, so the cost is paid on every workflow with no corresponding signal. Fixing the root causes converts a rubber-stamp step into a real gate; the alternative proposed in #163 (deletion) would save the cost but leave QA as a gap in the chain.

## User Story

As a developer running SDLC workflows, I want the QA skills to find real bugs, gaps, and risks in my PRs so that QA is a meaningful pre-merge gate rather than a rubber-stamp, and so that the workflow chain has an adversarial check that complements (rather than mirrors) the implementation and review steps that came before it.

## Motivation

Issue #170 (building on the diagnosis in #163) identifies three structural reasons the current QA skills cannot produce a real signal:

1. **No independent oracle.** `qa-verifier.md` defines verification as "use Read to open the file and confirm the described condition" / "use Grep to search for functions or patterns". Test plan entries are generated by the same model from the same requirements doc that drove implementation. The implementation was written to satisfy the spec; the verifier re-reads the spec and checks the implementation matches. Both sides share the ground truth, so verification collapses to a consistency check, not a correctness check.
2. **The Stop hook grades the string, not the truth.** `executing-qa/scripts/stop-hook.sh` blocks until the last message contains regex-matched phrases like `verdict.*pass`, `verification.*complete`, `all.*entries.*pass`, `reconciliation.*complete`. With no external signal available, the model is biased toward charitable resolution of every ambiguous entry.
3. **The plan-builder has the same closed loop.** `documenting-qa` compares a generated plan against the source requirements doc; both sides were generated from the same source, so completeness verification collapses to "did we copy FR-1 into a row?"

This redesign attacks all three:

- **Closed loop** → broken by forbidding the planning agent from reading the requirements doc during plan construction (FR-4). Reconciliation against the spec happens only at the end and the interesting signal is the *delta* (FR-5).
- **String grading** → replaced by artifact-based grading (FR-8, FR-9). The stop hook validates that tests were written, executed, and results produced — or that the run is a legitimate `EXPLORATORY-ONLY` fallback (FR-3).
- **Verification prompt** → replaced by an adversarial tester persona (FR-6) that probes failure modes the implementation likely did not anticipate.

## Architecture Overview

The redesign is organized as three independently-swappable layers so that future features (additional execution modes, additional personas) can extend a layer without restructuring the skill:

| Layer | Purpose | Extensibility point |
|---|---|---|
| Capability discovery | Detects what is testable in the consumer repo (test framework, runner scripts, package manager) | Add future execution modes (CLI, HTTP, library) by extending the discovery step |
| Persona prompt module | Supplies the tester-style prompt overlay | Add future personas (a11y, security, performance) by adding new modules the skill composes in |
| Executable-oracle runner | Writes and runs tests, collects results | Add future runners (CLI-exec, HTTP-exec, library-exec) that plug into the same runner interface |

The first cut ships:
- Capability discovery with detection for `vitest`, `jest`, `pytest`, and `go test`.
- One persona module: `qa` (generic adversarial tester).
- One runner: **test-framework mode** (writes tests in the detected framework and runs them).
- **`EXPLORATORY-ONLY` fallback** runner for repos where no framework is detected.

## Functional Requirements

### FR-1: Capability Discovery Step

At the start of every `documenting-qa` and `executing-qa` run, the skill must execute a capability-discovery scan of the consumer repo and persist a structured **capability report**. The report is consumed by both the planning prompt (to constrain what kinds of tests can be proposed) and the execution runner (to select the execution mode).

Minimum detection (first cut):

| Signal | Detection rule | Capability |
|---|---|---|
| `vitest` | `package.json` has `vitest` as a dep, or `vitest.config.{ts,js,mjs}` exists | `test-framework: vitest` |
| `jest` | `package.json` has `jest` as a dep, or `jest.config.{ts,js,mjs,json}` exists | `test-framework: jest` |
| `pytest` | `pyproject.toml` declares `pytest`, `pytest.ini` exists, or `tests/` contains `pytest`-shaped files | `test-framework: pytest` |
| `go test` | `go.mod` exists and any `*_test.go` file exists | `test-framework: go-test` |
| Package manager | `package-lock.json` / `yarn.lock` / `pnpm-lock.yaml` present (for the npm-family frameworks) | `package-manager: npm \| yarn \| pnpm` |
| Test command | `package.json` `"scripts"."test"` exists (for npm-family), or conventional runner invocation for others | `test-command: <string>` |

The capability report is written to a non-durable location (e.g. `/tmp/qa-capability-{ID}.json`) — it is recomputed at the start of each run so stale reports cannot mislead a subsequent invocation. The report must include at least:

```json
{
  "id": "FEAT-XXX",
  "timestamp": "2026-04-19T14:22:00Z",
  "mode": "test-framework",
  "framework": "vitest",
  "packageManager": "npm",
  "testCommand": "npm test",
  "language": "typescript",
  "notes": ["<any detection warnings>"]
}
```

When no supported framework is detected, the report sets `mode: "exploratory-only"` and `framework: null`, and the run proceeds per FR-3.

Detection order is deterministic: `vitest` → `jest` → `pytest` → `go test`. If two frameworks are detected (e.g., a JS repo with both `vitest` and `jest` installed), the first match wins and a note is appended to `notes`.

### FR-2: Executable Oracle Runner — Test-Framework Mode

When the capability report indicates `mode: "test-framework"`, `executing-qa` must:

1. Write one or more test files in the detected framework that exercise the adversarial scenarios from the test plan (FR-4).
2. Execute them via the detected `testCommand`.
3. Capture structured results (total, passed, failed, errored test names, duration, truncated failing-test output).
4. Persist the results to `qa/test-results/QA-results-{ID}.md` in the new structured format (FR-9).

Test files are written under a conventional QA-owned directory (e.g. `<project-test-root>/qa/` or the framework's standard test directory tagged with a QA prefix) and committed to the PR branch as part of the run's output. The specific directory convention is configurable per repo; the default is the framework's standard test directory with a `qa-` filename prefix.

The run's verdict is:

| Condition | Verdict |
|---|---|
| All written tests pass, and at least one adversarial scenario was covered | `PASS` |
| One or more written tests fail | `ISSUES-FOUND` |
| Test execution errors (runner crash, compile error) — not a test failure | `ERROR` |
| No tests were written for adversarial scenarios (empty plan, no applicable scenarios) | `EXPLORATORY-ONLY` (falls back to FR-3) |

The runner must not grade on the absence of findings alone: a `PASS` verdict is only valid when the run actually wrote and executed tests that exercised scenarios beyond what the implementation's own test suite already covered.

### FR-3: Exploratory-Only Fallback Runner

When the capability report indicates `mode: "exploratory-only"` (no supported framework detected), or when the run is unable to produce executable tests for any reason, the skill must:

1. Emit a structured exploratory review covering the adversarial dimensions from FR-6 (inputs, state transitions, environment, dependency failure, cross-cutting).
2. Label the run verdict `EXPLORATORY-ONLY` (not `PASS`, not `FAIL`).
3. Include a machine-readable reason in the results artifact explaining why no executable component ran:

   ```
   ## Exploratory Mode
   Reason: No supported test framework detected in consumer repo.
   Detection attempted: vitest, jest, pytest, go test.
   Consumer repo root: /path/to/repo.
   ```

4. Still require the agent to surface at least one plausible issue per applicable adversarial dimension, or explicitly justify why the dimension is not applicable to the change at hand. An empty exploratory review is a stop-hook failure (FR-8).

`EXPLORATORY-ONLY` runs count as completed for orchestrator state-transition purposes (the workflow advances) but are visually distinguished from `PASS` runs in the artifact and in any downstream reporting.

### FR-4: No-Spec-During-Planning (Planning Prompt Change)

`documenting-qa` must build its test plan **without reading the requirements document**. The planning agent's prompt receives only:

- **User-facing summary** — a 2-5 sentence description of what the feature claims to do, in user terms. Derived from the PR title + PR body (first paragraph) or, if the PR does not yet exist, from the feature's `User Story` section of the requirements doc. In the second case, only the `User Story` section is passed through — not the FR grid, not the ACs, not the edge-case list.
- **Code context** — the PR diff (when a PR exists) or the set of files changed on the feature branch since `main`.
- **Capability report** (FR-1) — framework, language, package manager.
- **Persona overlay** (FR-6) — the adversarial tester prompt module.

The planning agent's prompt must explicitly forbid reading `requirements/features/FEAT-*.md`, `requirements/chores/CHORE-*.md`, or `requirements/bugs/BUG-*.md` during plan construction. The forbid is enforced by the prompt text, not by tool restrictions — the requirements doc is still readable by later phases of the skill (for reconciliation, FR-5), it is only excluded from the planning phase.

The output test plan is organized by **use case and failure mode**, not by FR row. Suggested top-level sections:

```
# Test Plan: {ID}
## User Summary
## Capability Report
## Scenarios (by dimension)
### Inputs
### State transitions
### Environment
### Dependency failure
### Cross-cutting (a11y, i18n, concurrency, permissions)
## Non-applicable dimensions (with justification)
```

Every scenario must be labeled with:
- A priority (`P0` / `P1` / `P2`) reflecting the tester's judgment of likelihood × impact
- An expected execution mode (`test-framework` or `exploratory`)
- A brief description of the expected test shape ("call `foo(null)`, assert `TypeError`")

### FR-5: Reconciliation Delta Report

After the executable-oracle run completes, `executing-qa` must produce a **bidirectional delta** against the requirements document:

- **Coverage beyond requirements** — scenarios QA tested that the spec did not mention. These are flagged as `coverage-surplus` and may indicate either good QA diligence or unneeded scope.
- **Planning gaps against requirements** — FRs / ACs / edge cases the spec demanded that QA did not include in the test plan. These are flagged as `coverage-gap` and are a signal that either the plan was incomplete or the spec is over-detailed.

The delta appears as a section in the results artifact (FR-9) with the structure:

```
## Reconciliation Delta
### Coverage beyond requirements
- Scenario X (tested by qa-boundary-inputs.spec.ts:12) — not mentioned in spec
### Coverage gaps
- FR-3 acceptance criterion "validates unicode input" — no corresponding scenario in plan
- AC "handles concurrent requests" — no corresponding scenario in plan
### Summary
- coverage-surplus: N
- coverage-gap: M
```

The delta does **not** modify the requirements document. Document updates (AC checkoff, completion status, PR link, affected files) are handled by the independent bookkeeping step tracked in issue #169 and are explicitly out of scope for this feature.

### FR-6: Adversarial Tester Persona

The first persona module shipped by this feature is `qa` — a generic adversarial tester. The persona overlay is a prompt fragment that augments the planning and execution prompts and must cover the following dimensions explicitly:

| Dimension | Example probes |
|---|---|
| Inputs | Boundary values, unicode, empty/null, oversized payloads, malformed formats, injection attempts |
| State transitions | Cancel mid-flow, double-click, back/forward nav, stale tabs, interrupted network mid-submit |
| Environment | Offline, slow network, cold cache, permission-denied filesystems, missing env vars |
| Dependency failure | API 500s, timeouts, rate limits, DB disconnects, third-party-service outages |
| Cross-cutting | a11y (keyboard nav, screen reader, color contrast), i18n (RTL, non-ASCII), concurrency (two clients, race, shared state), permissions/authz |

The persona overlay must include an explicit **"empty findings is suspicious"** directive: if the plan or run produces zero issues for any applicable dimension, the agent must justify (in the artifact) why the dimension is not applicable. The stop hook (FR-8) enforces this: a plan or artifact that covers only one dimension without justifying the others fails validation.

### FR-7: Persona-Module Composition Slot

The skill architecture must be structured so that adding a new persona (e.g., `a11y`, `security`, `performance`) in a future feature does not require restructuring the skill. Concretely:

- Persona content lives in `plugins/lwndev-sdlc/skills/documenting-qa/personas/{name}.md` (and the matching location under `executing-qa/`).
- The skill's planning and execution prompts reference the persona by name (`--persona qa` default) and compose the persona file into the prompt at invocation time.
- The first cut registers only the `qa` persona. Adding future personas is a matter of dropping a new file into the `personas/` directory and documenting the trigger in the skill manifest; no changes to the runner, stop hook, or capability discovery.

The persona name is passed through the orchestrator as a plain string. The first cut does not expose a CLI flag for persona selection (the default `qa` is used); a future feature may add `--persona <name>`. This FR is satisfied by the directory structure and composition mechanism being in place — no persona-selection UI is required in the first cut.

### FR-8: Stop-Hook Rewrite

The existing `plugins/lwndev-sdlc/skills/executing-qa/scripts/stop-hook.sh` must be rewritten so it no longer regex-matches PASS phrases. The new hook validates:

1. A structured results artifact exists at `qa/test-results/QA-results-{ID}.md`.
2. The artifact contains a `## Verdict` section with one of the allowed values: `PASS`, `ISSUES-FOUND`, `ERROR`, `EXPLORATORY-ONLY`.
3. The artifact contains a `## Capability Report` section (populated from FR-1).
4. For `PASS` or `ISSUES-FOUND` verdicts: the artifact contains a `## Execution Results` section with machine-readable counts (`total`, `passed`, `failed`, `errored`) and the run's exit code. The counts and exit code must be consistent with the verdict.
5. For `EXPLORATORY-ONLY`: the artifact contains a `## Exploratory Mode` section with a `Reason:` line.
6. For any verdict: the artifact includes the per-dimension coverage from FR-6 or a justification for non-applicability.

A bare message containing "verdict: pass" with no artifact fails validation. An artifact that is present but malformed (missing sections, inconsistent counts) fails validation with a specific error pointing to the missing/inconsistent field.

A matching rewrite is required for `documenting-qa`'s stop hook; update per the same artifact-first approach (the planning stop hook should validate that a structured plan artifact exists at `qa/test-plans/QA-plan-{ID}.md` with the FR-4 sections populated, not that particular phrases are present).

### FR-9: Structured Results Artifact Format

`qa/test-results/QA-results-{ID}.md` must follow a versioned, structured format. First-cut schema:

```markdown
---
id: FEAT-XXX
version: 2
timestamp: 2026-04-19T14:22:00Z
verdict: PASS | ISSUES-FOUND | ERROR | EXPLORATORY-ONLY
persona: qa
---

## Summary
{One-line summary of the run}

## Capability Report
{Copied from FR-1}

## Execution Results
- Total: N
- Passed: N
- Failed: N
- Errored: N
- Exit code: 0 / N
- Duration: Xs
- Test files: [qa-inputs.spec.ts, qa-state.spec.ts, ...]

## Scenarios Run
{Per-scenario table: id, dimension, priority, result, link to test file}

## Findings
{Per-finding entry: severity, dimension, title, reproduction, evidence}

## Reconciliation Delta
{Per FR-5}

## Exploratory Mode
{Only for EXPLORATORY-ONLY runs, per FR-3}
```

The `version: 2` field distinguishes the new format from the pre-redesign format. Parsers must treat absence of the `version` frontmatter field as version 1 (the legacy format) and handle both during the transition.

`qa/test-plans/QA-plan-{ID}.md` gets an equivalent versioned format with the sections from FR-4.

### FR-10: Artifact Format Versioning

Existing `qa/test-results/QA-results-*.md` artifacts (the 34 files currently in the directory, all version 1) must be retained as a historical record and must not be deleted or migrated. The new `version: 2` format is clearly distinguished via the frontmatter field (FR-9).

Any downstream tooling that consumes QA artifacts must either gate on `version` or produce the same output for both. In practice, there is no downstream tooling that consumes these artifacts today; this FR is documentation-level (future consumers must honor the version field).

### FR-11: `reviewing-requirements` Test-Plan Reconciliation Mode Decision

The test-plan reconciliation mode of `reviewing-requirements` currently re-reviews the QA plan against the source requirements doc. Under the new design, the plan deliberately diverges from the requirements doc (FR-4), so "does the plan match the spec?" is no longer the right question to ask.

This FR requires an explicit documented decision:

1. **Option A — Repurpose the mode**: the mode re-reviews the plan to verify the FR-4 structural rules are followed (no spec rows copied in, dimensions covered, priorities assigned, etc.). The mode's prompt is rewritten accordingly.
2. **Option B — Remove the mode**: the orchestrator no longer invokes it (feature step 6, chore/bug step 4 drops the fork); the mode is deleted from `reviewing-requirements/SKILL.md`.

The decision must be recorded in the implementation plan's Phase 0 (or wherever the decision lands) and reflected in the acceptance criteria. Both options require updating `orchestrating-workflows/SKILL.md` chain tables.

### FR-12: Orchestrator Chain-Table Update

Depending on the FR-11 decision, the orchestrator's chain tables (`plugins/lwndev-sdlc/skills/orchestrating-workflows/SKILL.md`) may need updating:

- If FR-11 **Option A** (repurpose): no chain-length change; the step stays at feature 6 / chore 4 / bug 4 with the same skill name but updated semantics.
- If FR-11 **Option B** (remove): the feature chain becomes 5+N+4 (down from 6+N+4), and the chore/bug chains become 7 steps (down from 8). All downstream step numbers are renumbered, mirroring the pattern used by FEAT-017.

`CLAUDE.md` workflow-chain descriptions must be updated to reflect any step changes.

## Non-Functional Requirements

### NFR-1: Portability Across Consumer Repos

The redesigned skills ship in the `lwndev-sdlc` plugin and run inside arbitrary consumer repos. They must not assume:
- Any specific test framework
- A running dev server
- Network access to third-party services
- A UI layer
- A specific language or runtime

Capability discovery (FR-1) is the mechanism that enforces this: anything not detected is gracefully handled via the `EXPLORATORY-ONLY` fallback (FR-3).

### NFR-2: Graceful Degradation

Any failure in the capability-discovery, planning, or execution phases must produce an artifact — never a silent no-op. Specifically:

- Detection failure → `EXPLORATORY-ONLY` run with reason populated.
- Planning failure (agent returns empty plan) → `fail` state with an actionable message.
- Execution failure (runner crashes, test files fail to compile) → `ERROR` verdict with stack traces in the artifact.
- Persona file missing → skill aborts at load time with a clear error pointing to the expected file path.

### NFR-3: Backward Compatibility of Existing Artifacts

Existing `qa/test-results/QA-results-*.md` artifacts (34 files) must remain readable and must not be modified or deleted by the redesign. They are version 1 (implicit, no frontmatter `version` field); new artifacts are version 2. Any downstream code that reads these artifacts must honor the version field.

### NFR-4: Test Coverage

The redesign itself must include tests:
- Capability discovery unit tests for each supported framework (vitest, jest, pytest, go-test) and for the no-framework case.
- Stop-hook unit tests for each artifact-shape (well-formed PASS / ISSUES-FOUND / ERROR / EXPLORATORY-ONLY / malformed).
- Integration test covering a full `documenting-qa` → `executing-qa` round trip in a tiny fixture repo that exercises the test-framework mode.

### NFR-5: Smoke-Run Demonstration

At least one end-to-end smoke run on a recent real feature branch (e.g., FEAT-016 or FEAT-017) must produce a `FAIL` or `ISSUES-FOUND` verdict — demonstrating the new skills can produce non-PASS outcomes. The first cut of the first cut must be capable of saying "no". This is captured in the acceptance criteria as a demonstration artifact.

### NFR-6: Orchestrator Compatibility

The redesigned skills must remain invocable from the orchestrator without changes to the orchestrator's main-context-step calling pattern. Specifically:

- `documenting-qa` is still called from main context at feature step 5 / chore step 3 / bug step 3.
- `executing-qa` is still called from main context at feature step 6+N+3 / chore step 7 / bug step 7.
- The expected artifact paths (`qa/test-plans/QA-plan-{ID}.md` and `qa/test-results/QA-results-{ID}.md`) do not change.

If FR-11 Option B is chosen, the orchestrator's chain tables change per FR-12, but the main-context calling pattern for the two surviving steps stays the same.

## Edge Cases

1. **Consumer repo has two test frameworks installed** (e.g., vitest and jest): detection order is deterministic (per FR-1), first match wins, detection note recorded. User can override via config in a future feature; first cut does not expose an override flag.
2. **Consumer repo has a framework but no `test` script**: detection succeeds on the framework signal; `testCommand` defaults to the framework's standard invocation (`npx vitest run`, `npx jest`, `pytest`, `go test ./...`). The skill logs that it fell back to the default command.
3. **Consumer repo has a framework but tests fail to compile** (TypeScript errors in test files the skill wrote): verdict is `ERROR` with the compile error in the artifact; skill does not retry (retry is a future feature).
4. **PR does not yet exist** (executing-qa runs before PR creation in some manual invocations): the skill uses `git diff main...HEAD` + feature doc's `User Story` section as the code context and user summary. Logged as a diminished-signal scenario in the artifact.
5. **Feature branch is clean relative to main** (no diff): `executing-qa` cannot run meaningfully; returns `ERROR` with a specific reason ("no changes to test relative to main").
6. **Capability discovery detects a framework but no test directory**: skill creates the conventional directory (`__tests__/` for jest-family, `tests/` for pytest, `_test.go` files alongside source for go). The directory creation is committed with the test files.
7. **Requirements doc is absent** (rare; e.g., a manual orchestrator invocation): planning prompt receives only the PR diff + capability report; reconciliation delta (FR-5) is skipped with a note.
8. **Persona file is missing or malformed**: skill aborts at load time with a clear error; does not silently substitute a default persona.
9. **Agent fails to probe any applicable adversarial dimension**: stop hook (FR-8) fails validation; the run must either produce findings or justify non-applicability per dimension.
10. **Large consumer repo** (>10k files): capability discovery should complete in <5 seconds (NFR to be tuned during implementation). The detection does not read every file — it reads only the manifest files (`package.json`, `pyproject.toml`, `go.mod`, etc.) and globs for a small number of config files.

## Testing Requirements

### Unit Tests

- Capability discovery: one test per supported framework (vitest, jest, pytest, go-test), plus the no-framework case, plus each edge case from the list above (1, 2, 6).
- Stop-hook validator: one test per artifact shape (well-formed for each verdict, malformed variants, missing sections, inconsistent counts).
- Persona-module loader: persona present, persona missing, persona malformed frontmatter.
- Delta reporter: coverage-surplus only, coverage-gap only, both, neither.

### Integration Tests

- `documenting-qa` round-trip on a fixture repo with a vitest config → produces a `qa/test-plans/QA-plan-*.md` artifact with the FR-4 structure.
- `executing-qa` round-trip on the same fixture repo with a deliberately-broken feature → produces an `ISSUES-FOUND` verdict with failing-test output in the artifact.
- `executing-qa` round-trip on a repo with no test framework → produces an `EXPLORATORY-ONLY` verdict with a populated exploratory section.

### Manual / Smoke Testing

- Invoke the redesigned `executing-qa` against a real recent PR in this repo (e.g., FEAT-017's PR) and verify the verdict is produced, the artifact is version 2, and the reconciliation delta is populated.
- Deliberately break a file in an existing feature branch, invoke `executing-qa`, and verify the skill produces `ISSUES-FOUND` rather than `PASS` — the NFR-5 demonstration.

## Future Enhancements

The following are explicitly **out of scope** for this feature and tracked as follow-up issues (to be created as part of the implementation plan):

- **CLI-exec execution mode**: fuzz CLI entry points with adversarial inputs; assert on exit codes and stdout.
- **HTTP-server execution mode**: spin up a dev server, hit endpoints with curl/fetch, assert on responses.
- **Pure-library execution mode**: import-and-call scripts for repos that are libraries without a CLI or server.
- **Additional personas**: `a11y`, `security`, `performance` — each a separate follow-up feature.
- **Persona-selection CLI flag**: `--persona <name>` exposed from the orchestrator, allowing workflows to select a different persona.
- **Two-persona-per-run composition**: composing multiple personas (e.g., `qa + a11y`) into a single run.
- **Bookkeeping migration**: AC checkoff, completion status, affected-files trim, PR link — tracked in #169, ships independently.
- **Detection coverage expansion**: Additional frameworks beyond the first-cut four (e.g., mocha, rspec, phpunit, JUnit).
